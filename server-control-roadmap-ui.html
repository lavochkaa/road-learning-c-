UI-дорожная карта: C++ контроль серверов

Пошаговое обучение в блоках. Иди этап за этапом, и ты сам соберешь рабочее приложение без "вайбкодинга".

Формат: Beginner-friendly Стек: C++17 + SSH + JSON + Qt Результат: Консоль + GUI окно

Как идти по этой дороге

  • На каждом блоке делай 4 шага: что изучить, что написать руками, как проверить, критерий готовности.
  • Не переходи дальше, пока текущий блок не стабильно работает.
  • Держи фокус на понимании: сначала маленький рабочий шаг, потом расширение.

Что нужно знать перед стартом

База C++

  • Функции и `struct`
  • `std::vector`, `std::string`
  • Обработка ошибок (`try/catch`)

База Linux/SSH

  • Команды `ssh`, `sudo`, `uptime`
  • Ключи SSH и доступ без пароля
  • Понимание stdout/stderr

Инструменты

  • Компилятор C++17 (clang++/g++)
  • CMake (желательно)
  • Позже: Qt Creator + Qt 6 Widgets

Концепция системы

  • `reboot/shutdown` = через SSH
  • `power on` выключенного хоста = через API
  • UI не должен запускать SSH напрямую

Пошаговые блоки обучения

Блок 1. Консольное меню-скелет

Цель: научиться управлять потоком программы и выбором действий.

Что сделать
  • Цикл меню
  • Выбор сервера по индексу
  • Пока только fake-действия
Как проверить
  • `0` завершает
  • Неверный ввод не ломает программу
Готовность
  • Меню стабильно работает 5+ запусков подряд
while (true) {
  int action = -1;
  std::cout << "0-exit, 1-status, 2-reboot: ";
  if (!(std::cin >> action)) { std::cin.clear(); std::cin.ignore(10000, '\n'); continue; }
  if (action == 0) break;
  std::cout << "action executed: " << action << "\n";
}

Цель: отделить данные от интерфейса.

Что сделать

  • Печатать список красиво
Как проверить

struct Server { std::string name, host, user; int port = 22; };
std::vector<Server> servers = {
  {"web-1", "203.0.113.10", "automation", 22},
  {"db-1",  "203.0.113.20", "automation", 22}

for (size_t i = 0; i < servers.size(); ++i)
  std::cout << i << ") " << servers[i].name << " [" << servers[i].host << "]\n";

Блок 4. Reboot/Shutdown

Цель: добавить реальные управляющие действия.

Что сделать
  • Добавить команды `sudo reboot`
  • И `sudo shutdown -h now`
  • Печатать `exit code`
Как проверить
  • Только тестовый сервер
  • После reboot сервер возвращается в сеть
Готовность
  • Обе команды вызываются из твоего меню
CommandResult rebootServer(const Server& s) {
  return sshExec(s, "sudo reboot");
}
CommandResult shutdownServer(const Server& s) {
  return sshExec(s, "sudo shutdown -h now");
}

Блок 5. Архитектура (ServerManager)

Цель: не смешивать UI, SSH и бизнес-логику.

Что сделать
  • Интерфейсы `IExecutor`, `IPowerAdapter`
  • `SSHExecutor`, `MockPowerAdapter`
  • `ServerManager` как единая точка логики
Как проверить
  • UI вызывает только `ServerManager`
  • Подмена адаптера не ломает UI
Готовность
  • SSH-команд нет в UI-коде
class IExecutor { public: virtual ~IExecutor() = default;
  virtual CommandResult exec(const Server&, const std::string&) = 0; };
class IPowerAdapter { public: virtual ~IPowerAdapter() = default;
  virtual bool powerOn(const Server&) = 0; virtual bool powerOff(const Server&) = 0; };
class ServerManager {
 public: CommandResult status(size_t i); bool powerOn(size_t i);
};

Блок 7. Power On/Off адаптер

Цель: разделить SSH действия и питание инстанса.

Что сделать
  • Сначала `MockPowerAdapter`
  • Потом реальный адаптер провайдера
  • Работать через `instanceId`
Как проверить
  • Mock печатает корректный action
  • Реальный режим включает/выключает хост
Готовность
  • `power on/off` не зависит от SSH
class MockPowerAdapter : public IPowerAdapter {
 public:
  bool powerOn(const Server& s) override {
    std::cout << "[MOCK] power on: " << s.instanceId << "\n"; return true; }
  bool powerOff(const Server& s) override {
    std::cout << "[MOCK] power off: " << s.instanceId << "\n"; return true; }
};

Блок 8. Безопасность

Цель: не оставить критические дыры в админке.

Что сделать
  • Только SSH-ключи
  • Whitelist действий
  • Ограничить `sudoers`
Как проверить
  • Нельзя выполнить произвольную команду
  • Каждое действие логируется
Готовность
  • Есть минимальный аудит и контроль доступа
enum class Action { Status, Reboot, Shutdown };
std::string toCmd(Action a) {
  switch (a) {
    case Action::Status: return "uptime && hostname && whoami";
    case Action::Reboot: return "sudo reboot";
    case Action::Shutdown: return "sudo shutdown -h now";
  }
  return "";
}

Как проверить

Готовность

connect(ui->btnStatus, &QPushButton::clicked, this, [this]{
  int idx = ui->serverList->currentRow();
  if (idx < 0) return;
  auto res = manager_.status(static_cast<size_t>(idx));
  ui->log->append(QString::fromStdString(res.out));
});

Блок 10. Обработка ошибок и UX

Цель: сделать поведение понятным и устойчивым при сбоях.

Что сделать
  • Понятные сообщения: timeout/auth failed/unreachable
  • Проверка индексов и входных данных
  • Подтверждение перед reboot/shutdown
Как проверить
  • Сломай host или key path и проверь текст ошибки
  • UI/консоль остаются рабочими после ошибки
Готовность
  • Программа предсказуемо работает и в успешных, и в аварийных сценариях
try {
  auto res = manager_.status(idx);
  if (res.exitCode != 0) throw std::runtime_error(res.out);
} catch (const std::exception& e) {
  ui->log->append("Error: " + QString::fromStdString(e.what()));
}

Итоговая архитектура

  • UI Layer: ConsoleMenu или MainWindow (Qt).
  • Service Layer: `ServerManager`.
  • Infrastructure Layer: `SSHExecutor`, `PowerAdapter`.
  • Data Layer: `servers.json`.
Главный принцип: UI вызывает только `ServerManager`. Прямые SSH/API вызовы в UI не делай.

Контрольные точки (Milestones)

  • M1: консольное меню + fake действия.
  • M2: `status` через SSH работает.
  • M3: `reboot/shutdown` работают на тестовом сервере.
  • M4: архитектура разделена на слои.
  • M5: список серверов читается из JSON.
  • M6: `power on/off` работает через адаптер.
  • M7: Qt-окно с кнопками вместо консоли.
  • M8: обработка ошибок + базовая безопасность.

Правило обучения

  • 1) Пишешь сам.
  • 2) Запускаешь.
  • 3) Ломаешь тестом.
  • 4) Чинишь.
  • 5) Только потом переходишь дальше.

Как понять, что ты реально научился

  • Можешь объяснить, почему `power on` и `reboot` идут разными путями.
  • Можешь заменить UI (консоль/Qt), не меняя `ServerManager`.
  • Можешь добавить новый сервер через JSON без изменения кода.
  • Можешь описать риски безопасности и где ты их закрыл.
Учебное правило: сначала сделай минимально рабочий шаг, потом усложняй. Не перескакивай через блоки.
